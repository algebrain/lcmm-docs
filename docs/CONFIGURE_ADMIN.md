# configure — руководство для администратора

Эта инструкция объясняет, как подготовить конфигурацию для приложения или модуля, которые используют `configure`.

Главная мысль здесь такая:

1. у приложения может быть один общий конфиг;
2. в этом конфиге могут лежать и настройки самого приложения, и настройки отдельных модулей;
3. приложение затем само распределяет эти значения по нужным частям системы.

Если нужен взгляд со стороны разработчика, читайте также [`CONFIGURE.md`](./CONFIGURE.md).

## 1. Источники конфигурации и порядок применения

Порядок такой:

1. TOML-файл;
2. переменные окружения поверх файла.

Если путь к файлу не задан явно и не включен режим `:env-only? true`, библиотека пытается найти файл по умолчанию:

1. `<module-name>_config.toml`;
2. `config.toml`.

Режимы работы обычно бывают такими:

1. только переменные окружения (`:env-only? true`);
2. явный файл через `:config` и переменные окружения поверх него.

## 2. Как выглядит один общий конфиг приложения

Для небольшого приложения с несколькими модулями удобно иметь один общий файл.

Пример, похожий на `reference-app`:

```toml
[app]
name = "reference-app"

[http]
port = 3006
expose_correlation_headers = true

[guard]
mode = "fail-open"
rate_limit = 60
rate_window_sec = 60
login_window_sec = 300
login_max_failures = 20
ban_ttl_sec = 900

[accounts]
db_path = "./data/accounts.db"

[catalog]
db_path = "./data/catalog.db"

[booking]
db_path = "./data/booking.db"

[notify]
db_path = "./data/notify.db"

[audit]
db_path = "./data/audit.db"
```

Что важно понимать:

1. это один файл для всего приложения;
2. в нем рядом лежат и настройки HTTP, и настройки защитного слоя, и пути к данным модулей;
3. модуль не обязан видеть этот файл целиком в таком виде;
4. приложение может взять из него только нужные значения и передать модулю уже его часть.

## 3. Как TOML превращается в ключи

`configure` преобразует TOML-секции в плоские dot-ключи.

Для примера выше это будут, например:

- `app.name`
- `http.port`
- `http.expose_correlation_headers`
- `guard.mode`
- `guard.rate_limit`
- `accounts.db_path`
- `booking.db_path`

Ключи нормализуются к lower-case.

## 4. Переменные окружения

Шаблон имени переменной:

```text
MODULE__SECTION__KEY
```

Префикс строится из `module-name` так:

1. имя переводится в верхний регистр;
2. все небуквенно-цифровые символы заменяются на `_`;
3. затем добавляется `__`.

Например, `reference-app` превращается в префикс `REFERENCE_APP__`.

Если приложение читает общий конфиг с `:module-name "reference-app"`, то примеры будут такими:

```text
REFERENCE_APP__HTTP__PORT=9090
REFERENCE_APP__HTTP__EXPOSE_CORRELATION_HEADERS=true
REFERENCE_APP__GUARD__RATE_LIMIT=120
REFERENCE_APP__GUARD__BAN_TTL_SEC=1800
REFERENCE_APP__ACCOUNTS__DB_PATH=/var/lib/reference-app/accounts.db
REFERENCE_APP__BOOKING__DB_PATH=/var/lib/reference-app/booking.db
```

Пустые значения окружения не переопределяют файл.

## 5. Что администратору важно не перепутать

Нужно различать два уровня.

### 5.1 Что относится к приложению

Например:

- `http.port`
- `http.expose_correlation_headers`
- `guard.mode`
- `guard.rate_limit`

### 5.2 Что относится к отдельным модулям

Например:

- `accounts.db_path`
- `catalog.db_path`
- `booking.db_path`
- `notify.db_path`
- `audit.db_path`

Практический смысл такой:

1. администратор готовит один файл или один набор переменных окружения;
2. приложение само разбирает, какие значения оставить себе;
3. приложение само передает каждому модулю его часть.

## 6. Ограничение разрешенных ключей

Если в приложении задан `:allowed-keys`, администратор должен использовать только эти ключи в переменных окружения.
Все остальные переменные окружения с префиксом приложения или модуля будут проигнорированы.
На ключи, которые уже лежат в TOML-файле, это ограничение не распространяется.

## 7. Два типовых сценария

### 7.1 Только переменные окружения

- файлов конфигурации нет;
- все значения задаются через окружение;
- приложение запущено с `:env-only? true`.

### 7.2 Явный файл и переменные окружения

- есть TOML-файл;
- путь к нему явно задан в приложении;
- переменные окружения переопределяют нужные значения поверх файла.

## 8. Рекомендации по безопасности

1. Не храните секреты в TOML, если их можно задать через окружение.
2. Не размещайте конфиги в общедоступных каталогах.
3. Делайте имена секретных ключей понятными: `password`, `token`, `secret`.
4. Если приложение выводит конфиг в лог, оно должно делать это только через безопасный вывод с маскированием чувствительных значений.

Важно: `dump-config` рассчитан на вывод конфигурации в том виде, который возвращает `load-config`, то есть плоской карты dot-ключей. Не используйте его как универсальное средство очистки произвольных вложенных структур приложения.

## 9. Практический вывод

Для небольшого приложения с несколькими модулями обычно достаточно одного общего файла конфигурации.
Это не означает, что модули теряют свои границы.
Это означает только то, что администратору удобнее поддерживать один понятный источник значений, а приложение уже само распределяет их дальше.
