# CONTRACT_INVARIANTS

Этот документ фиксирует инварианты взаимодействия фронтенда с LCMM-бекендом.

Он отвечает на вопрос:

что фронтенд может считать устойчивым контрактом, даже если внутренняя реализация
бекенда меняется.

## Зачем это нужно

LCMM описывает много внутренних механизмов: HTTP-обвязку, трассировку, события,
реактивные последствия, защитный слой, realtime-доставку.

Фронтенду не нужно знать все эти внутренности в деталях.
Ему нужно понимать другое:

1. на какие свойства ответов и поведения можно опираться;
2. какие предположения безопасны;
3. какие предположения делать нельзя.

## 1. HTTP остается основным каналом чтения состояния

Для фронтенда это означает:

1. начальное состояние экрана читается по HTTP;
2. повторное чтение после потери realtime-соединения тоже делается по HTTP;
3. веб-сокет не считается заменой обычного API.

Практическое следствие:

1. экран должен уметь загрузиться без websocket;
2. экран должен уметь восстановиться после reconnect обычным HTTP-чтением;
3. websocket следует считать каналом доставки обновлений поверх HTTP, а не
   единственным источником истины.

## 2. У приложения есть единый контракт ошибок

Фронтенд может ожидать:

1. структурированное тело ошибки;
2. стабильное поле `code` для клиентской логики;
3. `message` как человекочитаемый fallback;
4. `details` как машинно-обрабатываемую структуру, если backend решил ее отдать.

Практическое следствие:

1. клиентская логика должна опираться на `code`, а не на текст `message`;
2. тексты ошибок нельзя считать стабильным API;
3. даже при отсутствии знакомых `details` UI должен уметь показать безопасный
   fallback.

## 3. У запроса и у всей цепочки есть разные идентификаторы

LCMM различает:

1. `x-correlation-id` — идентификатор всей цепочки целиком;
2. `x-request-id` — идентификатор конкретного HTTP-запроса.

Для фронтенда это значит:

1. оба значения полезны для отладки;
2. `x-request-id` обычно формируется сервером, а не клиентом;
3. `x-correlation-id` клиент передает только если это явно согласовано на уровне
   приложения.

Практическое следствие:

1. при ошибке полезно сохранять оба значения;
2. bug report или support payload должен включать оба идентификатора, если они
   доступны.

## 4. Backend сохраняет причинно-следственную связность

LCMM исходит из того, что backend может проследить цепочку:

1. HTTP-запрос;
2. корневое событие;
3. производные события;
4. аудит и логи.

Для фронтенда это важно не как внутренняя деталь, а как обещание:

1. по `correlation-id` backend-команда может найти всю цепочку;
2. один UI-сбой можно связать с несколькими backend-действиями без ручного
   угадывания.

## 5. Веб-сокет не обязан повторять внутренние события один к одному

Фронтенд не должен предполагать, что browser-facing websocket message:

1. равен payload внутреннего доменного события;
2. использует те же имена полей;
3. раскрывает весь внутренний смысл backend-события.

Фронтенд может предполагать только это:

1. приложение отдает согласованный внешний websocket-контракт;
2. этот контракт ориентирован на браузерный клиент;
3. если соединение потеряно, клиент должен уметь перечитать состояние по HTTP.

## 6. Ограничение частоты и деградация считаются штатными режимами

Фронтенд должен считать нормальными такие ответы:

1. `429`, когда сработало ограничение частоты или временная блокировка;
2. `503`, когда зависимость или защитный слой временно недоступны;
3. безопасные auth/security ошибки без раскрытия лишних деталей.

Практическое следствие:

1. UI не должен считать эти ответы "невозможными";
2. клиент должен уметь делать backoff;
3. клиент не должен пытаться угадывать внутреннюю причину по тексту ошибки.

## 7. Безопасные ошибки важнее подробных ошибок

В LCMM защита от утечки важнее удобства угадывания по текстам.

Фронтенд не должен ожидать, что:

1. auth-ошибка сообщит, существует ли пользователь;
2. отказ в подписке по websocket подробно объяснит внутреннюю причину;
3. error-body раскроет внутренние детали инфраструктуры.

Практическое следствие:

1. UI должен уметь работать с минимально достаточной ошибкой;
2. безопасность не должна строиться на надежде получить "подсказку" от сервера.

## Чего фронтенд не должен предполагать

Фронтенд не должен предполагать:

1. что точные тексты `message` стабильны;
2. что `details` всегда имеют одну и ту же форму для всех типов ошибок;
3. что websocket может заменить initial load и resync;
4. что auth-ошибки объяснят, существует ли пользователь;
5. что внутреннее доменное событие автоматически совпадает с браузерным
   сообщением;
6. что отсутствие `Retry-After` разрешает бесконечно повторять запрос без
   паузы.

## Минимальная рабочая модель для клиента

Если свести LCMM к одной короткой модели для фронтенда, она выглядит так:

1. читать состояние по HTTP;
2. обрабатывать ошибки по `status` и `code`;
3. сохранять `correlation-id` и `request-id` для диагностики;
4. считать websocket каналом обновлений поверх HTTP;
5. уметь переживать `429`, `503`, reconnect и повторное чтение состояния.
