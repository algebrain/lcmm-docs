# HTTP_AND_ERRORS

Этот документ описывает правила работы фронтенда с HTTP и ошибками в
LCMM-приложении.

Здесь собран и расширен frontend-oriented материал из HTTP и security docs LCMM.

## 1. Базовая модель ошибки

Ожидаемая форма error-body:

```json
{
  "code": "validation_failed",
  "message": "Validation failed",
  "details": {},
  "correlation-id": "c1a2...",
  "request-id": "r9f0..."
}
```

Практические правила:

1. Для клиентской логики ориентируйтесь на `code`, не на `message`.
2. `message` используйте как fallback-текст для человека.
3. `details` обрабатывайте как машинно-читаемую структуру.
4. Не завязывайтесь на точные тексты ошибок.

Граница этого раздела:

1. форма error-body и опора на `code` относятся к устойчивым ожиданиям;
2. конкретная клиентская организация сетевого слоя не фиксируется LCMM и
   остается выбором проекта.

## 2. Что считается устойчивым, а что нет

Фронтенд может считать устойчивыми:

1. наличие поля `code` для машинной логики;
2. наличие `message` как текста для человека;
3. возможность получить `details`, если сервер решил передать дополнительную
   структуру;
4. наличие trace id в headers или в error-body.

Фронтенд не должен считать устойчивыми:

1. точные формулировки `message`;
2. полную форму `details` для каждого вида ошибок;
3. внутренние тексты исключений и инфраструктурных сбоев.

## 3. `details` для ошибок валидации

Один из удобных клиентских подходов:

1. ошибка уровня формы показывается как общая ошибка формы;
2. ошибка уровня поля маппится к конкретному полю;
3. незнакомые поля в `details` не должны ломать UI.

Пример:

```js
function mapValidationError(errBody) {
  const details = errBody?.details ?? {};
  return {
    formError: details.form || null,
    fieldErrors: details.fields || {}
  };
}
```

Практическое правило:

1. если `details` понятны клиенту, их нужно использовать;
2. если `details` нет или структура незнакома, UI все равно должен иметь
   безопасный fallback по `code` и `message`.

## 4. `correlation-id` и `request-id` в HTTP-ответе

`lcmm-http` использует два идентификатора:

1. `x-correlation-id` для всей цепочки;
2. `x-request-id` для конкретного HTTP-запроса.

Кто формирует эти поля:

1. `x-request-id` формируется серверным middleware;
2. `x-correlation-id` сервер либо принимает из входящего запроса, либо
   генерирует сам;
3. фронтенд обычно не должен отправлять `x-request-id`;
4. `x-correlation-id` фронтенд может передавать только если это явно согласовано
   на уровне приложения.

Зачем это фронтенду:

1. прикладывать идентификаторы к bug report;
2. связывать UI-ошибку с backend-логами;
3. различать несколько параллельных запросов на одном экране.

Важно для браузера:

1. JavaScript увидит эти headers только если приложение открыло их через
   `Access-Control-Expose-Headers`;
2. practically useful минимум: `x-correlation-id, x-request-id`.

## 5. Как читать основные статусы

### `400` и `422`

Обычно это ошибки входных данных.

Фронтенду обычно полезно:

1. не повторять такой запрос автоматически;
2. показать ошибки формы или поля;
3. опираться на `code` и `details`, а не на сырой текст.

### `401`

Обычно это проблема аутентификации или сессии.

Фронтенду обычно полезно:

1. считать это сигналом "пользователь не аутентифицирован";
2. не пытаться угадывать по сообщению, существует ли пользователь;
3. переводить пользователя в поток повторного входа, если это соответствует
   приложению.

### `403`

Обычно это отказ в доступе.

Фронтенду обычно полезно:

1. считать, что запрос понят серверу, но не разрешен;
2. не ожидать подробного объяснения причины;
3. показывать безопасное сообщение без лишних деталей.

### `404`

Это нормальный бизнес-результат, а не обязательно авария.

Фронтенду обычно полезно:

1. различать "ресурс не найден" и "экран сломан";
2. показывать предсказуемое состояние отсутствующего ресурса;
3. не запускать бесконечные повторы по умолчанию.

### `409`

Это конфликт с текущим состоянием предметной области.

Обычно это означает:

1. состояние изменилось;
2. действие больше не применимо;
3. нужен повторный показ актуального состояния, а не слепой повтор.

Фронтенду обычно полезно:

1. объяснить пользователю, что действие конфликтует с текущим состоянием;
2. при необходимости перечитать актуальное состояние по HTTP.

### `429`

Это ограничение частоты или временная блокировка.

Фронтенду обычно полезно:

1. считать ответ штатным;
2. проверить `Retry-After`;
3. применить backoff;
4. не спамить повторными запросами без паузы.

### `503`

Это временная недоступность или деградация.

Фронтенду обычно полезно:

1. считать ситуацию временной, а не обязательно фатальной;
2. применять контролируемую стратегию повторов;
3. показывать пользователю понятное сообщение о временной недоступности.

## 6. Повторы запросов и `Retry-After`

Если пришел `429` или `503`:

1. сначала проверьте `Retry-After`;
2. если header есть, используйте его как базовую задержку;
3. если header нет, используйте fallback backoff, например exponential backoff
   с небольшим случайным разбросом.

Пример:

```js
function parseRetryAfterSeconds(headers) {
  const raw = headers.get("retry-after");
  const n = Number(raw);
  return Number.isFinite(n) && n > 0 ? n : null;
}

function computeDelayMs(attempt, retryAfterSec) {
  if (retryAfterSec != null) return retryAfterSec * 1000;
  const base = Math.min(1000 * 2 ** attempt, 15000);
  const jitter = Math.floor(Math.random() * 250);
  return base + jitter;
}
```

Практические правила:

1. не повторяйте автоматически `400`, `401`, `403`, `404` как будто это
   временный сетевой сбой;
2. `429` и `503` требуют паузы перед повтором;
3. даже для `503` число автоматических повторов должно быть ограничено.

Это хорошие базовые решения для большинства клиентов, но не обязательная
схема организации всех запросов в каждом приложении.

## 7. Приведение ошибок к единому виду на клиенте

Пример с `fetch`:

```js
export async function apiFetch(url, init = {}) {
  const res = await fetch(url, init);
  const correlationId = res.headers.get("x-correlation-id");
  const requestId = res.headers.get("x-request-id");

  if (res.ok) return res;

  let body = null;
  try {
    body = await res.json();
  } catch (_) {}

  const err = {
    status: res.status,
    code: body?.code ?? "unknown_error",
    message: body?.message ?? "Request failed",
    details: body?.details ?? null,
    correlationId: body?.["correlation-id"] ?? correlationId,
    requestId: body?.["request-id"] ?? requestId,
    retryAfterSec: parseRetryAfterSeconds(res.headers)
  };

  throw err;
}
```

Пример с Axios:

```js
import axios from "axios";

const api = axios.create({ baseURL: "/api" });

api.interceptors.response.use(
  (response) => response,
  (error) => {
    const res = error.response;
    if (!res) throw error;

    const data = res.data || {};
    const normalized = {
      status: res.status,
      code: data.code || "unknown_error",
      message: data.message || "Request failed",
      details: data.details || null,
      correlationId: data["correlation-id"] || res.headers["x-correlation-id"] || null,
      requestId: data["request-id"] || res.headers["x-request-id"] || null,
      retryAfterSec: Number(res.headers["retry-after"]) || null
    };

    throw normalized;
  }
);
```

Эти примеры показывают удобную технику, а не обязательную форму клиентского
слоя.

## 8. Публичный и диагностический режимы ответов

Для части технических endpoint-ов LCMM различает:

1. безопасный публичный ответ;
2. более подробный диагностический ответ.

Для фронтенда это означает:

1. нельзя рассчитывать, что production API всегда отдаст подробную диагностику;
2. UI должен корректно жить и с минимальным безопасным набором полей;
3. диагностические детали нельзя считать обязательной частью обычного
   пользовательского сценария.

## 9. Что прикладывать в bug report

При отправке backend-команде полезно прикладывать:

1. endpoint и HTTP method;
2. `status` и `code`;
3. `correlation-id` и `request-id`;
4. timestamp и timezone;
5. минимальный шаг воспроизведения.

Не прикладывайте:

1. access token;
2. cookie или session id;
3. чувствительные пользовательские данные без необходимости.
