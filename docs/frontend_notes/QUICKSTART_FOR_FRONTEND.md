# QUICKSTART_FOR_FRONTEND

Этот документ дает короткий вход для frontend-интеграции с LCMM-бекендом.

Его задача:

1. быстро отделить устойчивые backend-инварианты от необязательных клиентских
   приемов;
2. дать минимальную рабочую модель клиента для малого и среднего проекта;
3. показать, в какой документ идти дальше за деталями.

Этот текст не заменяет остальные документы каталога.
Он нужен как короткая рабочая памятка.

## 1. Что backend действительно гарантирует

Если смотреть на LCMM с точки зрения фронтенда, безопасно исходить из
следующего:

1. HTTP остается основным способом чтения состояния;
2. websocket дополняет HTTP, а не заменяет его;
3. у ошибок есть единый внешний контракт;
4. для клиентской логики надежнее опираться на `code`, а не на `message`;
5. `correlation-id` и `request-id` являются нормальной частью диагностики;
6. `401`, `403`, `429`, `503` считаются штатными ответами, а не невозможными
   случаями;
7. browser-facing websocket message не обязан совпадать с внутренним событием
   один к одному;
8. безопасные auth- и security-ошибки могут намеренно не раскрывать лишние
   детали.

## 2. Что frontend может безопасно предполагать

Для клиента обычно безопасны такие ожидания:

1. экран должен уметь загрузиться по HTTP без обязательной зависимости от
   websocket;
2. после потери websocket-соединения состояние можно восстановить повторным
   HTTP-чтением;
3. если подписка по websocket отклонена, это не обязательно означает поломку
   приложения;
4. если сервер дал минималистичную auth-ошибку, это может быть нормальным
   следствием политики безопасности;
5. если сервер прислал `429` или `503`, клиенту нужна пауза перед повтором;
6. если `details` отсутствуют или имеют незнакомую форму, UI все равно должен
   иметь безопасный запасной путь.

## 3. Чего не стоит делать

В интеграции с LCMM обычно не стоит:

1. строить клиентскую логику на точных текстах `message`;
2. считать websocket единственным источником истины для экрана;
3. ожидать, что reconnect сам по себе восстановит все пропущенные изменения;
4. ждать от auth-ошибки подсказки "пользователь существует" или
   "пользователь не существует";
5. автоматически повторять `401`, `403`, `404` так, будто это обычный
   временный сетевой сбой;
6. спешно повторять запрос после `429` без паузы;
7. передавать секреты в URL и query string;
8. прикладывать cookie, session id или token к bug report.

## 4. Минимальная рабочая модель клиента

Для малого или среднего проекта обычно достаточно такой опорной схемы:

1. читать начальное состояние по HTTP;
2. приводить ошибки к единому виду в одном месте клиентского слоя;
3. ветвить клиентское поведение по `status` и `code`;
4. сохранять `correlation-id` и `request-id` для диагностики;
5. использовать websocket как канал последующих обновлений;
6. после потери соединения перечитывать состояние по HTTP, если нужна точная
   текущая картина.

Это не единственная допустимая архитектура клиента.
Но это хороший базовый режим, который хорошо согласуется с LCMM и не
переусложняет проект.
