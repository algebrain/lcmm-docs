# REALTIME_AND_SYNC

Этот документ описывает клиентскую модель realtime в LCMM.

Главный принцип со стороны бекенда:

1. HTTP загружает состояние;
2. websocket доставляет обновления;
3. при потере websocket клиент восстанавливает согласованность через HTTP.

Это не означает, что у фронтенда есть ровно один допустимый способ построить
экран.
Это означает, что LCMM дает фронтенду устойчивую опору:

1. начальное и восстановительное чтение можно делать по HTTP;
2. websocket можно использовать как канал доставки последующих изменений;
3. потеря websocket не должна делать приложение логически неработоспособным по
   определению.

Граница этого документа:

1. инварианты со стороны бекенда здесь важнее, чем рекомендации по устройству
   клиентского кода;
2. описанные ниже стратегии являются опорными примерами, а не обязательной
   архитектурой экрана.

## 1. Инварианты со стороны бекенда

### 1.1 Веб-сокет не заменяет HTTP

В LCMM websocket нужен не для того, чтобы убрать обычное API.
Он нужен для push-доставки событий в браузер.

Для фронтенда это означает:

1. начальная загрузка экрана делается по HTTP;
2. websocket может подключаться до, после или рядом с initial load, если
   клиенту так удобнее;
3. если websocket временно недоступен, у фронтенда все равно остается
   корректный HTTP-путь чтения состояния.

Практическое следствие:

1. фронтенд не обязан строить экран вокруг постоянного сокет-соединения;
2. websocket можно использовать как ускоритель обновлений, а не как
   единственный способ увидеть состояние.

### 1.2 Внешнее websocket-сообщение может отличаться от внутреннего события

Приложение может:

1. убрать лишние поля;
2. переименовать поля в более удобную внешнюю форму;
3. добавить технические поля вроде `correlationId`;
4. сгруппировать внешнюю доставку по темам, удобным для клиента.

Фронтенд не должен предполагать:

1. что payload websocket равен внутреннему доменному payload;
2. что формат browser-facing сообщения будет совпадать с event-bus envelope;
3. что клиент видит все внутренние детали backend-события.

### 1.3 Потеря соединения не обязана сопровождаться догрузкой пропущенных событий

Со стороны LCMM безопасно предполагать:

1. reconnect сам по себе не гарантирует доставку всех пропущенных обновлений;
2. после reconnect клиент может восстановить согласованность через HTTP;
3. websocket не отменяет необходимость reread состояния, если клиенту нужна
   точная текущая картина.

### 1.4 Подписка и отказ в подписке являются штатными сценариями

Фронтенд может рассчитывать на то, что:

1. подписка является отдельным действием;
2. у этого действия может быть и успех, и отказ;
3. отказ не обязан раскрывать внутренние детали авторизации или структуры
   данных.

## 2. Практичные клиентские стратегии

Из этих инвариантов хорошо следует несколько практичных стратегий.
Они не обязательны, но обычно удобны.

### 2.1 Стратегия "снимок состояния + обновления"

Одна из самых естественных стратегий:

1. сначала прочитать текущее состояние по HTTP;
2. затем открыть websocket;
3. подписаться на нужные темы;
4. принимать последующие обновления;
5. после reconnect или сомнения в согласованности перечитать состояние по HTTP.

Эта стратегия удобна потому, что:

1. она переживает потерю соединения;
2. она не делает websocket единственным источником истины;
3. она лучше согласуется с архитектурой LCMM, где websocket является transport
   уровня приложения.

Но это именно удобная стратегия, а не единственно допустимая модель клиента.

### 2.2 Стратегия "действие по HTTP, обновление через push или повторное чтение"

Еще одна естественная стратегия:

1. пользователь вызывает HTTP-операцию;
2. HTTP дает подтверждение или ошибку;
3. websocket может доставить производное обновление;
4. если push не пришел или соединение упало, состояние можно восстановить
   повторным HTTP-чтением.

Эта стратегия тоже опирается на инварианты LCMM, но не навязывается как
единственный допустимый вариант.

## 3. Что именно может ожидать фронтенд от websocket-контракта

На уровне первой фазы websocket в LCMM фронтенд может ожидать сообщения вида:

1. `subscribed`;
2. `unsubscribed`;
3. `pong`;
4. `error`;
5. `event`.

Практический смысл:

1. клиенту не стоит придумывать собственную модель подтверждения подписки;
2. клиенту полезно явно обрабатывать и успех подписки, и отказ;
3. heartbeat и диагностика соединения должны опираться на `ping`/`pong`, если
   они используются приложением.

## 4. Что полезно учитывать при reconnect

Клиенту полезно помнить:

1. повторное подключение не гарантирует, что все пропущенные обновления будут доставлены;
2. после reconnect имеет смысл перечитать snapshot по HTTP;
3. если приложение само делает HTTP reread после закрытия соединения, это не
   случайная деталь, а правильный паттерн.

Это хорошо видно и в `booking-full`:

1. при закрытии websocket demo-страница делает повторное чтение `/bookings`;
2. затем пробует переподключиться;
3. тем самым сначала восстанавливает состояние, а затем возвращает push.

Практическое правило:

1. повторное подключение и повторное согласование состояния лучше проектировать
   как пару, а не как независимые вещи.

## 5. Подписки и отказ в подписке

Фронтенду полезно уметь:

1. явно отправлять подписку;
2. явно получать подтверждение подписки;
3. явно обрабатывать отказ.

Если подписка отклонена:

1. клиенту не стоит угадывать по тексту, существует ли чужая сущность или нет;
2. клиенту полезно воспринимать отказ как безопасный ответ без раскрытия деталей;
3. интерфейсу обычно достаточно единого понятного сообщения, а не сложных
   догадок по содержимому ошибки.

## 6. Что делать при ошибках входящих сообщений

Если приложение сообщает:

1. `invalid_message`;
2. `subscription_rejected`;
3. `connection_rejected`;

клиенту полезно:

1. журналировать это как техническое событие;
2. показать пользователю безопасное короткое сообщение;
3. не переходить к бесконечному циклу мгновенных повторов;
4. если соединение закрыто, переходить в сценарий повторного подключения и
   повторного согласования состояния.

## 7. Пример из `booking-full`

В `booking-full` эта модель выглядит так:

1. экран сначала читает `/bookings`;
2. затем открывает `/ws?user-id=...`;
3. потом подписывается на тему пользователя;
4. при `booking/created` получает browser-facing событие с `bookingId`,
   `userId`, `slotId` и `correlationId`;
5. при закрытии соединения перечитывает `/bookings` и делает reconnect.

Это хороший опорный пример для frontend-интеграции:

1. HTTP используется для snapshot;
2. websocket используется для push;
3. reconnect не считается достаточным без reread состояния.

Важно:

1. это удачный пример клиентской стратегии;
2. это не единственная допустимая frontend-архитектура поверх LCMM.

## 8. Частые ошибки на стороне фронтенда

1. Делать websocket единственным источником состояния экрана.
2. Считать reconnect достаточным без resync по HTTP.
3. Ожидать, что browser-facing message совпадает с внутренним событием 1:1.
4. Не обрабатывать отказ в подписке как штатный сценарий.
5. Строить UX так, будто push всегда приходит надежно и вовремя.

## 9. Минимальный практический вывод

Если сократить все до одной формулы:

1. LCMM дает фронтенду надежный HTTP-путь чтения состояния;
2. websocket дает путь для доставки последующих обновлений;
3. дальше фронтенд волен строить удобную ему клиентскую стратегию, пока он не
   делает websocket единственным обязательным источником истины.
