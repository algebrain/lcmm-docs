# SECURITY_AND_AUTH

Этот документ описывает последствия для фронтенда от политики безопасности LCMM.

Его задача не в том, чтобы пересказать внутреннее устройство защитного слоя.
Его задача в том, чтобы фронтенд понимал:

1. какие ответы и ограничения считать штатными;
2. какие детали сервер намеренно не раскрывает;
3. как строить клиентское поведение без конфликта с защитной моделью.

## 1. Ожидания от аутентификации

Для LCMM нормальная модель такая:

1. аутентификация и проверка сессии живут на уровне приложения;
2. бизнес-модули не должны дублировать проверку сессии каждый по-своему;
3. фронтенд взаимодействует с единым app-level поведением, а не с разными
   auth-правилами каждого модуля.

Для фронтенда это означает:

1. `401` следует считать app-level сигналом "нужно заново пройти auth flow";
2. не нужно ожидать, что каждый endpoint будет оформлять auth-ошибки по-своему;
3. экрану с защищенными действиями полезно иметь понятную реакцию на потерю
   аутентификации.

## 2. Защита от перебора важнее подробной ошибки

LCMM прямо требует anti-enumeration.

Это означает:

1. ошибки входа должны быть унифицированы;
2. сервер не должен раскрывать, существует ли пользователь;
3. различия в тексте и коде ошибки не должны помогать перебору учетных данных.

Для фронтенда это означает:

1. не стоит строить UI, который ждет от сервера подсказки "логин верный, пароль нет";
2. не стоит интерпретировать auth-ошибки как источник информации о наличии
   пользователя;
3. обычно уместно показывать единое безопасное сообщение о неуспешном входе.

## 3. `401`, `403` и `429` нужно считать штатными ответами

### `401`

Обычно означает:

1. пользователь не аутентифицирован;
2. сессия истекла;
3. нужен повторный вход.

Фронтенду обычно полезно:

1. запускать понятный сценарий повторной аутентификации;
2. не угадывать внутреннюю причину по тексту ошибки;
3. не делать бесконечные автоматические повторы того же запроса.

### `403`

Обычно означает:

1. доступ к действию запрещен;
2. запрос понят, но не разрешен политикой безопасности.

Фронтенду обычно полезно:

1. показывать безопасное сообщение о запрете действия;
2. не ожидать подробного объяснения причины;
3. не превращать `403` в бесконечный цикл переповторов.

### `429`

Обычно означает:

1. ограничение частоты;
2. временную блокировку;
3. срабатывание защитного слоя по IP или по частоте запросов.

Фронтенду обычно полезно:

1. считать такой ответ штатным;
2. применять backoff;
3. учитывать `Retry-After`, если он есть;
4. не ускорять блокировку агрессивными мгновенными повторами.

## 4. Ограничение частоты и временные блокировки

LCMM исходит из того, что repeated invalid requests могут привести к:

1. детекту подозрительной активности;
2. временному ограничению;
3. временной блокировке.

Для фронтенда это означает:

1. repeated invalid login attempts нельзя считать безобидным UX-циклом;
2. клиенту не стоит автоматически повторять запрос к endpoint сразу после `429`;
3. у формы входа или защищенного действия полезно предусмотреть короткую
   паузу перед следующей попыткой.

Практический вывод:

1. после `429` полезно временно блокировать повторную отправку на клиенте;
2. таймер ожидания лучше показывать пользователю, если есть понятная оценка;
3. даже если точного TTL нет, повтор должен идти не мгновенно.

## 5. Последствия сессионной модели и cookie

LCMM рекомендует app-level сессионный подход.

Для фронтенда это означает:

1. сессия может жить в cookie;
2. потеря или истечение сессии проявляется как `401`, а не обязательно как
   "сломанный backend";
3. state-changing операции могут зависеть от CSRF-защиты и общей app-level
   политики.

Практические правила:

1. не пытайтесь передавать секреты и session data в query string;
2. не включайте в bug report cookie и session id;
3. клиентскому состоянию полезно уметь переходить в режим "нужен повторный
   вход" без разрушения всего приложения.

## 6. Что фронтенд не должен передавать

С точки зрения безопасной интеграции фронтенд не должен:

1. передавать секреты в URL и query string;
2. прикладывать access token, cookie и session id в сообщения об ошибке;
3. логировать чувствительные payload целиком без необходимости;
4. строить клиентские обходы ограничений частоты.

Это особенно важно для:

1. браузерных ссылок;
2. аналитических и клиентских логов;
3. скриншотов и bug report.

## 7. Ожидания безопасности для websocket

Если приложение использует websocket, фронтенд должен исходить из того, что:

1. websocket endpoint принадлежит приложению, а не отдельному модулю;
2. handshake и проверка `Origin` живут на app-level;
3. каждый `subscribe` может отдельно проверяться на доступ;
4. отказ в подписке не обязан раскрывать подробности.

Для фронтенда это означает:

1. отказ в подписке следует считать штатным сценарием;
2. нельзя ожидать, что сервер объяснит, существует ли чужой ресурс или нет;
3. reconnect проходит ту же проверку доступа заново;
4. после reconnect нужно быть готовым и к повторному отказу.

## 8. Формы и защищенные экраны

Для форм входа, административных действий и защищенных экранов полезно
исходить из следующего:

1. для ошибки входа обычно уместен безопасный единый текст;
2. repeated failures могут привести к `429`;
3. защищенному экрану полезно уметь реагировать на `401` как на потерю
   состояния аутентификации;
4. отказ в доступе не должен ломать весь экран, если можно показать локальное
   состояние "доступ запрещен".

Практический вывод:

1. для `401`, `403`, `429` часто полезны разные состояния интерфейса;
2. но ни одно из них не должно зависеть от раскрытия внутренней причины сервером.

## 9. Частые ошибки на стороне фронтенда

1. Ожидать от auth-ошибки подробного объяснения.
2. Показывать разные тексты для "пользователь не существует" и "неверный пароль"
   на основе догадок.
3. Автоматически повторять `429` как обычный сетевой сбой.
4. Передавать секреты в URL.
5. Считать websocket-подписку всегда успешной и не проектировать отказ.

## 10. Минимальный практический вывод

Если сократить все до одной рабочей модели:

1. безопасность LCMM намеренно скрывает лишние детали;
2. `401`, `403`, `429` являются частью штатного поведения;
3. фронтенд должен строить UX на безопасных инвариантах, а не на ожидании
   подробных объяснений от сервера.
