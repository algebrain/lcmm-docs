# TRACING_AND_DEBUGGING

Этот документ объясняет, как фронтенд использует инварианты трассировки LCMM.

Здесь важно не внутреннее устройство `event-bus`, а то, как фронтенд:

1. связывает ошибку на экране с backend-трассой;
2. сохраняет полезные идентификаторы;
3. строит удобный сценарий диагностики для разработчика и поддержки.

## 1. Два идентификатора, а не один

В LCMM фронтенду важно различать:

1. `correlation-id` — идентификатор всей цепочки;
2. `request-id` — идентификатор конкретного HTTP-запроса.

Коротко:

1. `correlation-id` нужен, чтобы backend нашел всю причинно-связанную историю;
2. `request-id` нужен, чтобы найти именно этот HTTP-запрос среди других.

Практическое правило:

1. если доступны оба значения, сохраняйте оба;
2. не подменяйте одно другим;
3. не называйте их в UI просто "request id", если на самом деле речь идет о
   `correlation-id`.

## 2. Что происходит с точки зрения фронтенда

С точки зрения браузера полезная модель выглядит так:

1. пользователь делает действие;
2. фронтенд отправляет HTTP-запрос;
3. сервер добавляет или принимает `correlation-id`;
4. сервер формирует `request-id` для этого HTTP-вызова;
5. если запрос порождает события и реактивные последствия, они продолжают жить
   под тем же `correlation-id`;
6. по этому же `correlation-id` backend-команда может найти логи, аудит и
   производные события.

Для фронтенда это означает:

1. один и тот же `correlation-id` может относиться не только к ответу API, но и
   к последующим side effects;
2. если проблема проявилась "не сразу", `correlation-id` все равно остается
   полезным ключом для разбора.

## 3. Что показывать в пользовательском и диагностическом интерфейсе

Обычному пользователю не нужен технический шум на каждом экране.
Но при ошибке полезно иметь управляемый способ показать trace id.

Рекомендуемый подход:

1. в обычном UI показывать понятное сообщение;
2. в деталях ошибки, debug panel или expandable block показывать
   `correlation-id`;
3. `request-id` держать для DevTools, логов клиента и bug report;
4. если места мало, наружу в первую очередь показывать `correlation-id`.

Почему именно так:

1. `correlation-id` полезнее для backend-диагностики;
2. `request-id` более локален и чаще полезен разработчику, чем пользователю.

## 4. Что сохранять в bug report или обращении в поддержку

Минимальный practically useful набор:

1. endpoint и HTTP method;
2. `status` и `code`;
3. `correlation-id`;
4. `request-id`, если он доступен;
5. время ошибки и timezone;
6. краткие шаги воспроизведения.

Не прикладывайте:

1. access token;
2. cookie;
3. session id;
4. чувствительные данные пользователя без необходимости.

## 5. Что полезно логировать на стороне клиента

Если приложение ведет клиентские технические логи, полезно сохранять:

1. URL запроса;
2. HTTP method;
3. статус ответа;
4. `code` ошибки;
5. `correlation-id`;
6. `request-id`;
7. отметку времени;
8. имя экрана или пользовательского сценария.

Практическое правило:

1. логируйте достаточно, чтобы связать браузерную ошибку с backend-трассой;
2. не логируйте секреты и чувствительные payload целиком без необходимости.

## 6. Сетевой слой, консоль и DevTools

Для разработчика фронтенда полезно считать обязательной такую проверку:

1. при любой нестандартной ошибке посмотреть `status`, `code` и headers;
2. отдельно выписать `x-correlation-id` и `x-request-id`;
3. проверить, совпадает ли `correlation-id` в error-body и в headers, если он
   есть в обоих местах;
4. передать этот идентификатор дальше в bug report.

Практическое правило:

1. не ограничивайтесь текстом "Request failed";
2. trace id должен быть частью стандартной процедуры отладки.

## 7. Трассировка в websocket-потоке

В websocket-сценариях трассировка тоже остается полезной.

Если приложение доставляет browser-facing сообщения через websocket, фронтенду
полезно:

1. читать `correlationId`, если он есть в сообщении;
2. связывать push-событие с предшествующим HTTP-действием, если это один и тот
   же пользовательский сценарий;
3. сохранять `correlationId` websocket-сообщения в debug log клиента.

Практическая граница:

1. фронтенд не должен предполагать, что у каждого websocket-сообщения всегда
   будет одинаковая форма трассировки;
2. но если `correlationId` есть, его нужно использовать как такой же ключ для
   backend-диагностики.

## 8. Частые ошибки на стороне фронтенда

1. Путать `request-id` и `correlation-id`.
2. Сохранять только текст ошибки и терять trace id.
3. Искать причину только в одном HTTP-ответе, хотя проблема уже ушла в
   реактивную цепочку.
4. Не показывать никакой технический идентификатор даже в debug UI.
5. Логировать сырые секреты вместе с трассировочными полями.

## 9. Минимальный практический вывод

Если сократить все до одного рабочего правила:

1. при любой значимой ошибке фронтенд должен уметь достать и сохранить
   `correlation-id`;
2. при наличии сохранять и `request-id`;
3. считать эти значения стандартной частью диагностики, а не редкой опцией "на
   случай больших проблем".
