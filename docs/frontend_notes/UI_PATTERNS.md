# UI_PATTERNS

Этот документ превращает инварианты LCMM в практические UI-паттерны.

Он отвечает не на вопрос "как backend устроен внутри", а на вопрос:

как строить клиентские экраны и обработчики действий так, чтобы они хорошо
согласовывались с LCMM-бекендом.

Важно:

1. это не контракт LCMM;
2. это не список обязательных правил для каждого фронтенда;
3. это набор практичных приемов, которые обычно хорошо работают поверх уже
   описанных инвариантов взаимодействия.

Если конкретный продукт, экран или клиентская платформа требуют другого
решения, это не считается нарушением LCMM само по себе.
Нарушением будет только опора на предположения, которые противоречат backend
инвариантам.

## 1. Паттерн отправки формы

Базовый flow для формы:

1. пользователь отправляет данные;
2. UI показывает состояние "идет запрос";
3. при успехе экран обновляется или делает переход;
4. при ошибке UI ветвится по `status` и `code`, а не по тексту ошибки.

Практическое правило:

1. клиентскому обработчику отправки часто удобно получать уже приведенную к
   единому виду ошибку;
2. форме обычно невыгодно разбирать сырой HTTP-ответ в каждом месте отдельно.

## 2. Структурированное отображение ошибок валидации

Если сервер вернул `details`, полезно разделять:

1. общую ошибку формы;
2. ошибки отдельных полей;
3. fallback-сообщение, если структура `details` не распознана.

Один из удобных подходов:

1. field-level ошибки маппить на поля;
2. form-level ошибку показывать над формой или под заголовком;
3. незнакомые ошибки не выбрасывать, а аккуратно сводить к общему сообщению.

## 3. Паттерн для `401`

Для `401` часто удобен такой ход:

1. остановить текущий protected action;
2. перевести экран в состояние "нужен повторный вход";
3. после успешной аутентификации решить, нужно ли повторять действие вручную.

Практическое правило:

1. не повторяйте автоматически чувствительные действия после `401` без явного
   контроля;
2. отличайте "нужно заново войти" от "все приложение сломано".

## 4. Паттерн для `403`

Для `403` часто удобен такой ход:

1. показать состояние "доступ запрещен";
2. не маскировать `403` под сетевую ошибку;
3. не предлагать бессмысленный бесконечный retry.

Это особенно полезно для:

1. административных экранов;
2. действий с ролями;
3. подписок и защищенных websocket-сценариев.

## 5. Паттерн для `409`

Для `409` часто удобен такой ход:

1. объяснить, что действие конфликтует с текущим состоянием;
2. перечитать актуальные данные;
3. перестроить экран на основе новой версии состояния.

Это лучше, чем:

1. бесконечно повторять тот же запрос;
2. оставлять пользователя с устаревшей локальной картиной.

## 6. Паттерн для `429` и временного охлаждения

Для `429` часто полезно делать так:

1. временно отключать кнопку повторной отправки;
2. если есть `Retry-After`, показывать ориентир по времени;
3. если `Retry-After` нет, использовать краткий локальный cooldown;
4. не скрывать, что ограничение пришло с сервера.

Практический смысл:

1. такой UI помогает не усиливать защитное ограничение;
2. пользователь видит, что проблема временная, а не случайная поломка.

## 7. Паттерн для `503` и деградации

Для `503` часто полезно различать:

1. единичный временный сбой действия;
2. более широкую деградацию экрана или приложения.

Один из удачных вариантов:

1. показать понятное временное сообщение;
2. разрешить ограниченный повтор;
3. если экран критичен, предложить ручное обновление состояния;
4. не разрушать уже загруженные данные без необходимости.

## 8. Паттерн "снимок состояния + push"

Для экранов с режимом реального времени один из самых удобных приемов такой:

1. получить snapshot по HTTP;
2. подключить websocket;
3. применять входящие обновления к уже загруженному состоянию;
4. при потере соединения перечитать snapshot по HTTP.

Практическое правило:

1. websocket ускоряет обновление экрана;
2. HTTP восстанавливает согласованность.

Но фронтенд не обязан буквально строить каждый экран именно так.
Это полезный базовый шаблон, а не жесткое ограничение.

## 9. Паттерн после reconnect

После повторного подключения часто полезно делать не только новое соединение,
но и повторное согласование состояния:

1. открыть новое соединение;
2. заново подписаться;
3. перечитать snapshot;
4. только после этого считать экран снова согласованным.

Это особенно полезно там, где:

1. события могли быть пропущены;
2. состояние экрана зависит от нескольких источников;
3. пользователь ожидает точную текущую картину.

Если конкретный экран может безопасно жить без повторного чтения, фронтенд
вправе выбрать более легкую стратегию.

## 10. Паттерн для debug-элементов в production UI

Даже если продуктовый интерфейс минималистичен, иногда полезно иметь
управляемый способ показать технические сведения:

1. раскрывающийся блок "техническая информация";
2. `correlation-id`;
3. `request-id`, если он доступен;
4. `status` и `code`.

Это помогает:

1. пользователю быстро передать данные в поддержку;
2. команде быстрее найти проблему;
3. не заставлять разработчика просить скриншот всей консоли.

## 11. Паттерн для клиентских логов и support flow

Если приложение собирает клиентские технические события, полезно связывать:

1. имя пользовательского сценария;
2. URL и метод запроса;
3. `status` и `code`;
4. `correlation-id`;
5. время ошибки.

Практическое правило:

1. путь обращения в поддержку полезно встраивать в продуктовый сценарий
   ошибки, а не держать только в голове разработчика.

## 12. Частые ошибки на стороне UI

1. Разбирать поведение сервера по тексту `message`.
2. Не различать `401`, `403`, `409`, `429`, `503` как отдельные UI-состояния.
3. Считать websocket единственным источником истины экрана.
4. После повторного подключения не перечитывать снимок состояния.
5. Не сохранять trace id для диагностики.

## 13. Минимальный практический вывод

Если сократить все до одной рабочей формулы:

1. формы опираются на `code` и `details`;
2. фронтенд может удобно опираться на модель "HTTP-снимок + websocket-обновления";
3. ошибки безопасности и ограничения считаются штатными UI-состояниями;
4. trace id является частью нормального пользовательского и support flow.
