# Browser API Checks For `reference-app`

Этот документ описывает полную последовательность ручных проверок `reference-app`
из адресной строки браузера.

Он рассчитан на запуск в режиме `--reset`, чтобы все шаги опирались на одно и то
же начальное состояние.

## 1. Подготовка

Запустите сервер:

```bash
bb start.bb --reset
```

Если порт не переопределялся, дальше используется:

```text
http://localhost:3006
```

Если сервер запущен с другим портом, замените `3006` на свой порт во всех
адресах ниже.

## 2. Что считается исходным состоянием

После запуска в режиме `--reset` предполагается:

1. существуют пользователи `u-admin` и `u-alice`;
2. существуют слоты `slot-09-00`, `slot-10-00`, `slot-11-00`;
3. списки бронирований, уведомлений и аудита пусты;
4. guard еще не блокировал ни один IP.

## 3. Порядок проверок

Последовательность ниже специально построена так, чтобы:

1. сначала проверить пустое стартовое состояние;
2. потом пройти сценарий создания бронирования;
3. затем проверить связанные изменения в `notifications` и `audit`;
4. в конце отдельно проверить сценарий guard.

## 4. Проверка app-level маршрутов

### 4.1 `/healthz`

Откройте:

```text
http://localhost:3006/healthz
```

Ожидаемый результат:

1. HTTP `200`;
2. тело показывает, что процесс жив;
3. в браузере вы увидите небольшой ответ со статусом `ok`.

### 4.2 `/readyz`

Откройте:

```text
http://localhost:3006/readyz
```

Ожидаемый результат:

1. HTTP `200`;
2. приложение считается готовым;
3. проверка `read-provider registry` проходит успешно.

### 4.3 `/metrics`

Откройте:

```text
http://localhost:3006/metrics
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ текстовый, в формате Prometheus;
3. в тексте должны встречаться метрики HTTP-сервера, например
   `http_server_requests_total`.

## 5. Проверка `accounts`

### 5.1 `/accounts/me`

Откройте:

```text
http://localhost:3006/accounts/me?user-id=u-alice
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. в ответе есть пользователь `u-alice`;
4. поле `:login` равно `"alice"`;
5. поле `:password` в ответ не попадает.

### 5.2 `/accounts/users/:user_id`

Откройте:

```text
http://localhost:3006/accounts/users/u-admin
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. в ответе есть пользователь `u-admin`;
4. логин равен `"admin"`.

### 5.3 Отрицательная проверка `accounts`

Откройте:

```text
http://localhost:3006/accounts/users/missing-user
```

Ожидаемый результат:

1. HTTP `404`;
2. текстовый ответ `user not found`.

## 8. Проверка сценария создания бронирования

### 8.1 Успешное создание

Откройте:

```text
http://localhost:3006/bookings/actions/create?slot-id=slot-09-00&user-id=u-alice
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. в ответе есть новый `:id` бронирования;
4. `:slot_id` равен `"slot-09-00"`;
5. `:user_id` равен `"u-alice"`;
6. `:status` равен `"created"`.

Важно:

1. сохраните значение `:id` из ответа;
2. оно понадобится для следующей проверки.

### 8.2 Получение созданного бронирования по id

Подставьте `BOOKING_ID` из предыдущего шага:

```text
http://localhost:3006/bookings/BOOKING_ID
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. `:id` совпадает с `BOOKING_ID`;
4. `:slot_id` равен `"slot-09-00"`;
5. `:user_id` равен `"u-alice"`.

### 8.3 Список бронирований после создания

Откройте:

```text
http://localhost:3006/bookings
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. в списке ровно одно бронирование.

### 8.4 Список бронирований по пользователю

Откройте:

```text
http://localhost:3006/bookings?user-id=u-alice
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. в списке есть созданное бронирование `u-alice`.

## 11. Проверка guard-сценария

Этот сценарий удобнее всего проверять в одном и том же окне браузера, чтобы
запросы шли с одного и того же клиентского адреса.

### 11.1 Успешный demo-login

Откройте:

```text
http://localhost:3006/auth/demo-login?login=alice
```

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. есть `:ok true`;
4. внутри `:user` находится пользователь `alice`.

### 11.2 Ошибка входа

Откройте:

```text
http://localhost:3006/auth/demo-login?login=missing
```

Ожидаемый результат:

1. HTTP `401`;
2. ответ в формате EDN;
3. `:code` равен `"invalid_credentials"`.

### 11.3 Повторная ошибка входа и блокировка

Сразу откройте тот же адрес еще раз:

```text
http://localhost:3006/auth/demo-login?login=missing
```

Ожидаемый результат:

1. HTTP `429`;
2. ответ показывает, что IP временно заблокирован или ограничен.

### 11.4 Проверка, что блокировка уже действует

Сразу после этого откройте:

```text
http://localhost:3006/auth/demo-login?login=alice
```

Ожидаемый результат:

1. HTTP `429`;
2. успешный логин пока не проходит, потому что guard уже заблокировал адрес.

### 11.5 Снятие блокировки

Откройте:

```text
http://localhost:3006/ops/guard/unban?ip=203.0.113.10
```

Если вы проверяете сценарий именно через встроенные тестовые вызовы приложения,
там используется `203.0.113.10`.
При реальной ручной проверке адрес может отличаться.

Практическое правило:

1. если блокировка не снимается этим адресом, проверьте фактический клиентский IP
   по логам сервера;
2. затем подставьте его в `ip=...`.

Ожидаемый результат:

1. HTTP `200`;
2. ответ в формате EDN;
3. `:ok` равен `true`.

### 11.6 Повторная успешная проверка после снятия блокировки

Откройте снова:

```text
http://localhost:3006/auth/demo-login?login=alice
```

Ожидаемый результат:

1. HTTP `200`;
2. успешный вход снова работает.

## 12. Когда сценарий нужно начинать заново

Если вы хотите снова пройти весь документ с теми же ожидаемыми результатами,
проще всего:

1. остановить сервер;
2. запустить его заново через `bb start.bb --reset`.

Это снова приведет систему к каноническому начальному состоянию.
